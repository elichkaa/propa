## The Problem

A data structure may have **invariants** — properties that must always hold (e.g., a rational number's denominator is never zero and the fraction is in lowest terms). If users can construct values directly, they can violate these invariants.

## Solution: Private Constructors + Smart Constructors

1. Mark the raw constructor as `private` — only accessible within the defining module.
2. Expose a **smart constructor** that enforces invariants before creating the value.

````lean
structure Rational where  
private mk' :: -- hidden outside this module  
num : Int  
den : Nat
````

The smart constructor `Rational.mk` handles:
- Zero denominator → defaults to `0/1`
- GCD reduction → always lowest terms
- Sign normalization → sign always on numerator

## Access Control in Lean

| Modifier | Visibility |
|---|---|
| (none) | Public — accessible everywhere |
| `private` | Only within the **same module/file** |
| `protected` | Only within the **same namespace** (must use qualified name elsewhere) |

> [!important]
> `private` in Lean is **module-level**, not namespace-level. Code in the same file can still access private definitions (even if it's in a different namespace). True hiding requires separate files.

## General Principle

The combination of:
- **Opaque types** (hidden constructors)
- **Smart constructors** (validated creation)
- **Interface operations** (type class instances for `Add`, `Mul`, etc.)

ensures that **every value of the type satisfies its invariants**, no matter what external code does. This is the functional programming equivalent of encapsulation in OOP.

> [!tip]
> If all construction and manipulation goes through validated functions, invariants hold by construction — no runtime checks needed in consuming code.
