## The Idea

Instead of testing specific inputs, **generate thousands of random inputs** and check whether the property holds for all of them. If any input violates the property, report it as a counterexample.

This is a middle ground between unit testing (weak guarantees, easy) and formal proofs (strong guarantees, hard).

## Using `plausible` in Lean

The `plausible` tactic auto-generates random inputs and checks the property:

```lean
example (xs : List α) : (reverse xs).length = xs.length := by
  plausible

example (xs : List α) : reverse (reverse xs) = xs := by
  plausible
```

If no counterexample is found, the test passes (but this is not a proof — just high confidence).

## Configuration

Control the number of tests and input sizes:

```lean
example (xs : List α) : reverse (reverse xs) = xs := by
  plausible (config := {numInst := 1000, maxSize := 1000})
```

Enable trace output to see which inputs were generated:

```lean
set_option trace.plausible.success true
```

## What It Can Test

`plausible` works for any property whose input types have:
- A random generator (`Arbitrary` instance)
- A way to simplify counterexamples (`Shrinkable` instance)

Built-in types (`Nat`, `Int`, `String`, `List`, `Bool`, etc.) have these instances already.

## Relationship to Formal Proofs

| Approach | Guarantee | Effort |
|---|---|---|
| Unit tests | Specific inputs only | Low |
| Property-based testing | Many random inputs | Medium |
| Formal proof | All inputs, unconditionally | High |

Lean supports all three. Property-based testing is the practical sweet spot for most development — if you later need certainty, you can upgrade a `plausible` check to a formal `theorem`.
