## What is a Tail Call?

A **tail call** is a function call that is the **last thing** a function does — there's no remaining work after the call returns. Compare:

```lean
-- NOT tail-recursive: after the recursive call, we still need to add 1
def len (xs : List α) : Nat :=
  match xs with
  | [] => 0
  | _ :: xs => 1 + len xs      -- len returns, THEN we add 1

-- Tail-recursive: the recursive call IS the final result
def lenAux (xs : List α) (acc : Nat) : Nat :=
  match xs with
  | [] => acc
  | _ :: xs => lenAux xs (acc + 1)   -- nothing happens after this call
```

## Why Tail Calls Matter

When a call is in tail position, the compiler can optimize aggressively:

| Optimization | What it does | Benefit |
|---|---|---|
| **Stack frame reuse** | No need to keep the caller's frame — reuse it for the callee | Constant stack space, no stack overflow |
| **Loop transformation** | A tail-recursive function compiles into a simple loop | Eliminates function call overhead entirely |

A non-tail-recursive function with depth n needs O(n) stack frames. A tail-recursive one needs O(1).

## How to Recognize Tail Recursion

Ask: "after the recursive call returns, is there anything left to do?"

| Pattern | Tail recursive? | Why |
|---|---|---|
| `1 + f xs` | ❌ | Still need to add 1 after `f` returns |
| `f xs (acc + 1)` | ✅ | `f` is the last operation |
| `(f xs) ++ [x]` | ❌ | Still need to append after `f` returns |
| `f xs (x :: acc)` | ✅ | `f` is the last operation |

## Same Complexity, Different Constants

Both `len` and `lenAux` are O(n). The difference is in constant factors and stack usage:

- `len` builds up n stack frames, then unwinds them adding 1 each time
- `lenAux` uses a single stack frame rewritten n times, accumulating the result as it goes

For very large inputs, the non-tail version may crash with a stack overflow while the tail version runs fine.

> [!summary]
> A tail call is a call with nothing left to do after it. The compiler can turn tail-recursive functions into loops — same asymptotic complexity but better constant factors and O(1) stack usage.
