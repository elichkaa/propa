## Why Interpreters?

Interpreters provide a **formal definition of a language's semantics** — they specify exactly how each construct evaluates. They're also practical for REPLs, scripting, and rapid prototyping (no compilation delay).

## The Language

A small functional language with integers, booleans, conditionals, and first-class functions:

```lean
inductive Expression where
  | intLiteral (n : Int)
  | boolLiteral (b : Bool)
  | add (e1 e2 : Expression)
  | multiply (e1 e2 : Expression)
  | less_than (e1 e2 : Expression)
  | ifThenElse (cond eThen eElse : Expression)
  | var (name : String)
  | lambda (name : String) (body : Expression)
  | apply (func arg : Expression)
```

## Values

Everything an expression can evaluate to:

```lean
inductive Value where
  | int (i : Int)
  | bool (b : Bool)
  | lambda (name : String) (body : Expression)
```

Helper extractors panic on type mismatch (convention: `!` suffix means "may crash"):

```lean
def Value.asInt!    : Value → Int
def Value.asBool!   : Value → Bool
def Value.asLambda! : Value → String × Expression
```

## The Substitution-Based Interpreter

Interpreters are `partial` because termination depends on the input program (which may loop forever).

Evaluation rules:

| Expression | Evaluation |
|---|---|
| Literal | Return corresponding `Value` directly |
| Arithmetic / comparison | Evaluate both operands, apply operator |
| `ifThenElse` | Evaluate condition, then evaluate the chosen branch |
| `lambda` | Return as a `Value.lambda` (unevaluated body) |
| `apply f arg` | Evaluate `f` to a lambda, evaluate `arg`, **substitute** arg into body, evaluate result |
| `var x` | Should have been eliminated by substitution — panic if reached |

```lean
partial def interp! : Expression → Value
  | .intLiteral i => .int i
  | .apply funexp arg =>
      let (name, body) := (interp! funexp).asLambda!
      let argval := interp! arg
      interp! (substitute body name argval)
  | .var name => panic! s!"Unbound variable {name}"
  -- ... other cases follow the same pattern
```

### Performance Problem

On each function application, `substitute` traverses the entire body — even if the variable doesn't appear in it. With nested applications, the body gets traversed repeatedly.
