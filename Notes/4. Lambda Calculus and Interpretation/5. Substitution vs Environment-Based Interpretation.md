## The Problem with Substitution

In the substitution-based interpreter, every `apply` triggers a full traversal of the function body to replace the parameter. This means:

- **Two traversals per application**: one for substitution, one for interpretation
- **Redundant work**: if the variable isn't even used in the body, the traversal is wasted
- Deeply nested applications cause repeated re-traversal

## The Solution: Environments

Instead of rewriting the expression, maintain a **map from variable names to values** and look up variables when needed:

```lean
abbrev Environment := Std.HashMap String Value
```

### What Changes

Most evaluation rules stay identical — just thread `env` through. The key difference is function application:

**Before (substitution):**
```lean
| .apply funexp arg =>
    let (name, body) := (interp! funexp).asLambda!
    let argval := interp! arg
    interp! (substitute body name argval)    -- traverses body twice
```

**After (environment):**
```lean
| .apply funexp arg =>
    let (name, body) := (interpEnv! env funexp).asLambda!
    let argval := interpEnv! env arg
    let env' := env.insert name argval       -- O(log n) insert
    interpEnv! env' body                     -- single traversal
```

Variable lookup replaces substitution:

```lean
| .var name =>
    match env.get? name with
    | some v => v
    | none   => panic! s!"Unbound variable: {name}"
```

### Trade-off

| | Substitution | Environment |
|---|---|---|
| Application cost | O(body size) per call | O(log n) insert + lookup |
| Simplicity | Closer to formal semantics | Closer to real implementations |
| Correctness | Needs careful capture-avoidance | Needs careful scoping (see [[Dynamic vs Lexical Scoping]]) |
