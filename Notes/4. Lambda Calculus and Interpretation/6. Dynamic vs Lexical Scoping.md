## The Bug

The naive environment-based interpreter has a subtle problem:

```lean
def plus := λ x. λ y. x + y
def inc  := plus 1           -- should "remember" x = 1
def prog := λ x. inc x       -- binds x = 41

interpEnv! ∅ (prog 41)       -- expected: 42, actual: 82
```

What happened:
1. `prog 41` binds `x = 41` in the environment
2. Inside `prog`, `inc` is called, which calls `plus`
3. `plus` looks up `x` — finds `x = 41` (from `prog`), not `x = 1` (from `inc`)
4. Result: `41 + 41 = 82`

## Dynamic vs Lexical Scoping

| | Dynamic scoping | Lexical scoping |
|---|---|---|
| Variable lookup | Most recent binding on the **call stack** | Binding where the function was **defined** |
| `x` in `plus` resolves to | Whatever `x` was last bound to by any caller | The `x` that was in scope when `plus` was created |
| Predictability | Fragile — behavior changes depending on call context | Predictable — behavior depends only on definition site |
| Used in | Early Lisps, some config languages | All modern languages (Lean, Haskell, Python, JS, etc.) |

## The Fix: Closures

A **closure** is a lambda bundled with the environment that existed when it was defined:

```lean
inductive Value where
  | int (n : Int)
  | bool (b : Bool)
  | closure (name : String) (body : Expression) (env : Environment)
```

### What Changes in the Interpreter

**Lambda evaluation** — capture the current environment:

```lean
| .lambda name body => .closure name body env    -- snapshot env at definition time
```

**Application** — use the closure's captured environment, not the caller's:

```lean
| .apply funexp arg =>
    let (name, body, funenv) := (interpEnv! env funexp).asClosure
    let argval := interpEnv! env arg
    let env' := funenv.insert name argval    -- extend the CLOSURE's env, not caller's
    interpEnv! env' body
```

The critical line is `funenv.insert` — we extend the environment **from where the function was defined**, not from where it's being called.

### The Fix in Action

```lean
#eval interpEnv! ∅ inc
-- closure "y" (x + y) {x → 1}     ← x=1 is captured!

#eval interpEnv! ∅ (prog 41)
-- 42  ✅
```

When `inc` is created, it captures `{x → 1}`. When later called inside `prog` where `x = 41`, the closure ignores the caller's `x` and uses its own `{x → 1}`.

> [!summary]
> - **Dynamic scoping**: variables resolve based on the call stack — fragile and surprising
> - **Lexical scoping**: variables resolve based on definition site — predictable and standard
> - **Implementation**: store a **closure** (lambda + captured environment) instead of a bare lambda. On application, extend the closure's environment, not the caller's.
