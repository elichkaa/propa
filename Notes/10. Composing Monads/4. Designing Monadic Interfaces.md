## The Principle

Instead of writing code against a **specific** monad (`StateT Int (ExceptT String IO)`), define **abstract effect interfaces** as type classes. The implementation can then be swapped without changing any business logic.

This is the same idea as type classes for data structures (e.g., `Queue`), applied to effects.

## Defining Effect Interfaces

Each group of related effects becomes its own type class, parameterized over an abstract monad `m`:

```lean
class DirectoryEffects (m : Type → Type) where
  homeDir : m System.FilePath
  getCurrentDir : m System.FilePath
  changeDir : System.FilePath → m Unit

class ErrorEffects (m : Type → Type) where
  throwCommandFormat : String → m α
  handleCommandFormatError : m α → (String → m α) → m α
  throwExit : m α

class VariableEffects (m : Type → Type) where
  getVar : String → m (Option System.FilePath)
  setVar : String → System.FilePath → m Unit

class LiftIO (m : Type → Type) where
  liftIO : IO α → m α
```

## Writing Logic Against Interfaces

All application code is written with **type class constraints**, never mentioning the concrete monad:

```lean
variable {m : Type → Type}
  [Monad m] [DirectoryEffects m] [ErrorEffects m] [LiftIO m]

def processCommand (cmd : Command) : m Unit := do
  match cmd with
  | .cd path =>
    let abs ← resolvePath path
    DirectoryEffects.changeDir abs
  | .exit =>
    ErrorEffects.throwExit
  | .pwd =>
    let dir ← DirectoryEffects.getCurrentDir
    LiftIO.liftIO (IO.println dir)
```

This function works with **any** monad that provides directory effects, error effects, and I/O lifting. It doesn't know or care whether that's `StateT` + `ExceptT` + `IO`, or something else entirely.

## Providing the Implementation

Define a concrete state, error type, and monad stack, then implement each interface:

```lean
structure ShellState where
  dir      : System.FilePath
  vars     : HashMap String System.FilePath
  backward : List System.FilePath
  forward  : List System.FilePath

abbrev M := StateT ShellState (ExceptT Error IO)

instance : DirectoryEffects M where
  homeDir := IO.currentDir
  getCurrentDir := return (← get).dir
  changeDir dir := modify fun st =>
    { st with dir := dir, backward := st.dir :: st.backward, forward := [] }

instance : ErrorEffects M where
  throwCommandFormat msg := throw (Error.commandFormat msg)
  -- ... other operations
```

## Benefits

| Benefit                    | How                                                             |
| -------------------------- | --------------------------------------------------------------- |
| **Testability**            | Swap `IO` for a mock monad in tests                             |
| **Separation of concerns** | Business logic doesn't know about `StateT`/`ExceptT`            |
| **Flexibility**            | Change the monad stack without touching command implementations |
| **Readability**            | Type constraints document exactly which effects a function uses |

## The Pattern in Summary

```
                    ┌─────────────────────┐
                    │  Effect Interfaces  │  (type classes)
                    │  DirectoryEffects   │
                    │  ErrorEffects       │
                    │  VariableEffect     │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │ Application  │  │  Helpers     │  │  Commands    │
    │ (run loop)   │  │ (paths,      │  │ (cd, ls,     │
    │              │  │  parsing)    │  │  find, ...)  │
    └──────────────┘  └──────────────┘  └──────────────┘
              │                │                │
              └────────────────┼────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │  Concrete Instance  │  (monad stack + instances)
                    │  M = StateT S       │
                    │      (ExceptT E IO) │
                    └─────────────────────┘
```

All arrows point **down** — application code depends only on interfaces, never on the concrete monad. The concrete implementation is plugged in at the bottom.

> [!summary]
> Define effects as type classes. Write all logic against those classes. Provide one concrete monad stack that implements them all. This gives you the modularity of dependency injection, powered by the type system — no runtime overhead, no frameworks.
