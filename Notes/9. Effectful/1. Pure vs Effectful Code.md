## Why Purity Matters

A **pure function** always returns the same output for the same input and has no side effects. This enables:

- **Equational reasoning**: you can substitute a function call with its result without changing program behavior
- **Easier testing**: no environment, state, or interaction order to worry about
- **Local reasoning**: understanding a function only requires reading its definition, not tracking global state

## The Problem

Real programs *need* effects — reading files, printing output, making network calls. If effects can happen anywhere, purity is lost entirely: you can never be sure a function is safe to refactor or reorder.

## The Solution: Effects in the Type System

Pure FP languages track effects **in types**. A function's type tells you whether it's pure or effectful:

| Type | Meaning |
|---|---|
| `String → String` | Pure — guaranteed no side effects |
| `String → IO Unit` | Effectful — performs I/O when executed |
| `String → Except Error Argument` | Effectful — may fail with an error |

> [!key idea]
> Effects are not banned — they're **quarantined**. You always know where effectful code begins and ends by reading types. Pure code stays pure, and effectful code is explicitly marked.

This means if you see a function with type `List Nat → Nat`, you know with certainty it doesn't read files, print to the console, or throw exceptions. The type is a **contract**.
