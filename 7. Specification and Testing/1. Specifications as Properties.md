## The Problem with Unit Tests

Unit tests check specific inputs:

```lean
#assert reverse [1,2,3] == [3,2,1]
```

This proves the function works for `[1,2,3]`. It says nothing about `[1,2,3,4]`, empty lists, lists with duplicates, or the millions of other possible inputs. Edge cases and subtle bugs slip through.

## What is a Specification?

A **specification** describes the intended behavior of a function as a set of **properties** — universal statements that should hold for *all* valid inputs, not just a handful of examples.

### Example: Specifying `reverse`

Without a spec, "correctness" is undefined. Here are four properties that together pin down what `reverse` should do:

| Property | Statement |
|---|---|
| Preserves length | `length (reverse xs) = length xs` |
| Involution | `reverse (reverse xs) = xs` |
| Singleton identity | `reverse [x] = [x]` |
| Distributes over append | `reverse (xs ++ ys) = reverse ys ++ reverse xs` |

## Writing Properties in Lean

Properties are expressed as `Prop` — logical propositions, not executable booleans:

```lean
abbrev reverse_same_length (xs : List α) : Prop :=
  (reverse xs).length = xs.length

abbrev reverse_involution (xs : List α) : Prop :=
  reverse (reverse xs) = xs

abbrev reverse_append (xs ys : List α) : Prop :=
  reverse (xs ++ ys) = reverse ys ++ reverse xs
```

Note: `=` (propositional equality) not `==` (computational boolean equality).

### Example: Specifying sorting

A correct sorting function must satisfy **two** properties:

| Property | Meaning |
|---|---|
| **Sorted output** | For all valid indices `i < j`, element at `i` ≤ element at `j` |
| **Permutation** | The output contains exactly the same elements as the input (same counts) |

Either property alone is insufficient:
- `fun xs => []` produces sorted output but isn't a permutation
- `fun xs => xs` is a permutation but isn't necessarily sorted

> [!important] Specifications Catch Design Errors
> Writing properties forces you to think precisely about what your function should do. Often the process of *writing* the spec reveals bugs or ambiguities before you even test.
