## The Problem

`plausible` needs to know how to generate random values for every type in your property. Built-in types work out of the box. Custom types need custom generators.

```lean
-- This fails: plausible doesn't know how to generate BinaryTree values
example (t : BinaryTree Int) (a : Int) : (t.insert a).sum = t.sum + a := by
  plausible   -- "Failed to create a testable instance"
```

## Defining `Arbitrary` (Random Generation)

Implement the `Arbitrary` type class. Use `Gen.sized` to control tree depth and `Gen.frequency` to choose between constructors with weights:

```lean
instance [Arbitrary α] : Arbitrary (BinaryTree α) where
  arbitrary := Gen.sized go
  where
    go : Nat → Gen (BinaryTree α)
    | 0 => return BinaryTree.empty
    | n+1 => Gen.frequency (default := return BinaryTree.empty)
        [ (1, return BinaryTree.empty)                -- weight 1: empty
        , (3, do                                       -- weight 3: node
            let elem ← arbitrary
            let left ← go (n / 2)
            let right ← go (n / 2)
            return BinaryTree.node elem left right)
        ]
```

Key design choices:
- **Size parameter** controls depth — prevents infinitely large trees
- **Halving** (`n / 2`) at each level keeps trees manageable
- **Weights** control the distribution — here nodes are 3× more likely than empty at each level

## Defining `Shrinkable` (Counterexample Simplification)

When a counterexample is found, it's often unnecessarily complex. `Shrinkable` suggests simpler versions to test. The framework repeatedly shrinks until it finds the **smallest** failing input:

```lean
instance : Shrinkable (BinaryTree α) where
  shrink
  | .empty => []                         -- can't simplify empty
  | .node _ left right => [left, right]  -- try just the subtrees
```

Without shrinking, you might get a counterexample with 50 nodes. With shrinking, it gets reduced to 2-3 nodes — much easier to debug.

## Constraining Random Inputs

Sometimes properties have preconditions (e.g., "for valid indices `i < j < list.length`"). Generating random `i` and `j` independently means most pairs won't satisfy the precondition — the test passes vacuously without actually checking anything.

### Solution: Bounded Generators

Define a wrapper type that generates values within a specific range:

```lean
structure Bounded (lower upper : Nat) where
  num : Nat

instance : Arbitrary (Bounded lower upper) where
  arbitrary := do
    let size ← Gen.getSize
    if upper - lower <= 0 then
      Except.error (GenError.genError "impossible to generate value")
    else
      Bounded.mk <$> Gen.chooseNatLt lower upper (by grind)
```

Now use it in properties to ensure preconditions are always met:

```lean
example (l : List Int)
    (i : Bounded 0 (l.length - 1))
    (j : Bounded (i.num + 1) l.length) :
  l.quicksort.isSorted i.num j.num := by plausible
```

Every generated `i` and `j` is guaranteed to be a valid index pair — no wasted tests.

## Summary

| Instance | Purpose | When you need it |
|---|---|---|
| `Arbitrary α` | Generate random values of type `α` | Custom data types in properties |
| `Shrinkable α` | Simplify counterexamples | Always (for readable failures) |
| Bounded wrappers | Constrain inputs to valid ranges | Properties with preconditions |

> [!summary]
> For custom types: implement `Arbitrary` (how to generate) and `Shrinkable` (how to simplify). For constrained inputs: wrap parameters in bounded types that only generate valid values. This ensures property-based testing actually exercises the interesting cases.
