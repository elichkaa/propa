## The Observation

Even `map` and `filter` still explicitly recurse. Many functions on lists share an even deeper pattern:

```
match lst with
| [] => <base value>
| hd :: tl => <combine hd with (recurse on tl)>
```

`sum`, `hash`, `partition`, `length`, `map`, `filter` — they all follow this template. The only things that change are the **base value** and the **combine function**.

## fold Captures This Pattern

```lean
def fold (init : β) (combine : α → β → β) (lst : List α) : β :=
  match lst with
  | [] => init
  | hd :: tl => combine hd (fold init combine tl)
```

| Parameter | Role |
|---|---|
| `init` | What to return for the empty list (replaces `[]`) |
| `combine` | How to merge the current element with the result from the tail (replaces `::`) |

### Intuition: Constructor Replacement

Folding **replaces constructors with functions**:

```
cons 1 (cons 2 (cons 3 []))
  ↓ fold with init=0, combine=(+)
 (+) 1 ((+) 2 ((+) 3  0))
  = 6
```

Every `cons` becomes `combine`, every `[]` becomes `init`.

## Examples

```lean
-- sum: replace [] with 0, cons with +
def sum := fold 0 (· + ·)

-- length: replace [] with 0, cons with "ignore element, add 1"
def length := fold 0 (fun _ acc => 1 + acc)

-- partition: accumulate two lists
def partition (p : α → Bool) :=
  fold ([], []) (fun x (ts, fs) =>
    if p x then (x :: ts, fs) else (ts, x :: fs))
```

None of these functions explicitly recurse — `fold` handles all the recursion.

## Deriving fold for Any Data Type

The fold function can be **mechanically derived** from any algebraic data type:

### Recipe

1. List each constructor's type
2. Replace every occurrence of the recursive type with a fresh type `β`
3. Each transformed constructor becomes a parameter of `fold`
4. The body matches on constructors and applies the corresponding parameter, recursing on sub-structures

### Example: Binary Tree

```
Constructors:
  empty : BinaryTree α                         → fempty : β
  leaf  : α → BinaryTree α                     → fleaf  : α → β
  node  : BinaryTree α → BinaryTree α → ...    → fnode  : β → β → β
```

```lean
def BinaryTree.fold (fempty : β) (fleaf : α → β) (fnode : β → β → β)
    : BinaryTree α → β
  | empty       => fempty
  | leaf x      => fleaf x
  | node l r    => fnode (l.fold fempty fleaf fnode) (r.fold fempty fleaf fnode)
```

Using it:

```lean
-- height: empty=0, leaf=1, node=1+max of children
def height := BinaryTree.fold 0 (fun _ => 1) (fun l r => 1 + Nat.max l r)

-- scaffold: keep structure, erase leaves
def scaffold := BinaryTree.fold empty (fun _ => empty) (fun l r => node l r)
```

### Example: Route (5 constructors)

```
destination name       → fdestination : String → β
unreachable            → funreachable : β
turn dir rest          → fturn : String → β → β
sequence r1 r2         → fsequence : β → β → β
alternative r1 r2      → falternative : β → β → β
```

Each constructor maps to one parameter. The fold body applies the matching parameter and recurses on sub-routes.

## Why Fold Matters

- **Eliminates boilerplate recursion**: you specify *what* to compute, not *how* to recurse
- **Works for any algebraic data type**: lists, trees, routes — the derivation is mechanical
- **Proves a deep connection**: fold is the **universal eliminator** for a data type. Any function defined by structural recursion can be expressed as a fold

> [!summary]
> Fold replaces constructors with functions. Derive it mechanically from any data type's constructors. It captures **all** structural recursion in a single higher-order function.
