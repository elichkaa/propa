## The General Rule for Structural Recursion

When a function uses structural recursion, the number of recursive calls equals the size of the input. The total cost is:

```
O(n × k)
```

where `n` = input size and `k` = cost of work done per recursive step.

## Examples

### Length — O(n)

```lean
def List.len (xs : List α) : Nat :=
  match xs with
  | [] => 0
  | _ :: xs => 1 + len xs
```

One recursive call per element. Each step does a pattern match and an addition — both O(1). Total: **O(n × 1) = O(n)**.

### Append — O(n)

```lean
def List.app (xs ys : List α) : List α :=
  match xs with
  | [] => ys
  | x :: xs' => x :: app xs' ys
```

Recurses on the **first** list only. Each step does a pattern match and a cons — both O(1). Total: **O(n)** where n = length of `xs`.

### Reverse — O(n²) ⚠️

```lean
def List.rev (xs : List α) : List α :=
  match xs with
  | [] => []
  | x :: xs => (rev xs) ++ [x]
```

One recursive call per element (n total). But each step calls `++`, which takes O(length of first argument). The first argument is `rev xs`, which has length proportional to `n`. So `k = O(n)`.

Total: **O(n × n) = O(n²)**.

For a 1000-element list, that's ~1,000,000 operations. This doesn't scale.

## The Fundamental Problem

Functional lists are **singly linked** — you can only efficiently add elements to the front (O(1) cons). Appending to the end requires traversing the entire list (O(n)). Algorithms that repeatedly append to the end tend to be quadratic.

## Can We Do Better?

The standard library's `List.reverse` runs in **O(n)**. The technique: **accumulators** and **tail recursion** (see [[Accumulators]] and [[Tail Calls and Tail-Call Optimization]]).

> [!important]
> Pure functional programming doesn't have destructive pointer updates like imperative code. But with the right techniques (accumulators, tail recursion, efficient data structures), you can match imperative performance for most algorithms.
