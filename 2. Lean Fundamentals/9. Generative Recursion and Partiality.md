## What is Generative Recursion?

In **structural recursion**, each recursive call is on a direct sub-part of the input (e.g., the tail of a list). In **generative recursion**, the recursive calls are on **new values computed from the input** — values that aren't direct sub-structures.

## Key Difference

| | Structural | Generative |
|---|---|---|
| Recurse on | Sub-part of input (`xs` from `x :: xs`) | Newly computed value (`less`, `greaterEq`) |
| Termination | Guaranteed by structure | Must be argued separately |
| Lean accepts? | Automatically | Requires `partial` or a termination proof |

## Example: Quicksort

```lean
partial def quicksort [Ord α] (list : List α) : List α :=
  match list with
  | [] => []
  | pivot :: tail =>
    let (less, greaterEq) := tail.partition (fun x => compare x pivot == .lt)
    quicksort less ++ [pivot] ++ quicksort greaterEq
```

`less` and `greaterEq` are generated by partitioning — they aren't the `tail` of the original list. Lean can't automatically see that they're smaller, so the function needs `partial`.

## Example: Collatz Sequence

```lean
partial def collatz (n : Nat) : Nat :=
  match n with
  | 0 => 0
  | 1 => 1
  | _ => if n % 2 == 0 then collatz (n / 2) else collatz (3 * n + 1)
```

The recursive argument `3 * n + 1` is *larger* than `n`. Whether this always terminates is an open problem in mathematics — Lean obviously can't prove it.

## Example: Breadth-First Search

```lean
partial def bfs (graph : HashMap Nat (List Nat))
    (queue : List Nat) (visited : HashSet Nat) (result : List Nat) : List Nat :=
  match queue with
  | [] => result
  | node :: rest =>
    if visited.contains node then bfs graph rest visited result
    else
      let newQueue := rest ++ graph.getD node []
      bfs graph newQueue (visited.insert node) (result ++ [node])
```

The queue can grow on each step (adding neighbors), so the input isn't structurally shrinking.

## The `partial` Keyword

By default, Lean requires **all functions to be total** — they must terminate for all inputs. This is checked at compile time.

`partial` tells Lean: "trust me, I know this might not be provably terminating, but I want to use it anyway."

| | Total (default) | `partial` |
|---|---|---|
| Termination | Proven by Lean | Programmer's responsibility |
| Used in proofs | Yes | No — `partial` functions are opaque to the prover |
| When to use | Structural recursion, simple patterns | Generative recursion, complex algorithms |

> [!warning]
> `partial` is not unsafe — the function will still be well-typed. But Lean can't reason about it in proofs, so prefer structural recursion when possible.

> [!summary]
> Structural recursion follows the data's shape and always terminates. Generative recursion computes new inputs and may require `partial`. Both are essential — structural for safety, generative for expressiveness.
