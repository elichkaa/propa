## The Problem with Exceptions

In imperative languages, any function can throw an exception ‚Äî there's no way to tell from a type signature whether a call might fail. This breaks equational reasoning and makes code harder to follow.

## Except as Typed Failure

`Except E Œ±` represents a computation that either:
- **Succeeds** with a value of type `Œ±`
- **Fails** with an error of type `E`

It's just a data type with two constructors:

```
Except.ok    : Œ± ‚Üí Except E Œ±
Except.error : E ‚Üí Except E Œ±
```

Unlike unchecked exceptions, the possibility of failure is **visible in the type**. 
A function `String ‚Üí Except Error Argument` tells you upfront it can fail.

## Syntactic Sugar

Inside a `do` block, `Except` supports familiar-feeling syntax:

| Syntax         | Meaning                                                                        |
| -------------- | ------------------------------------------------------------------------------ |
| `return x`     | Succeed with value `x` (wraps in `Except.ok`)                                  |
| `throw err`    | Fail with error `err` (wraps in `Except.error`)                                |
| `let x ‚Üê comp` | Run `comp`; if it succeeds bind to `x`; if it fails **propagate error upward** |

> [!important] Error Escalation
> `let x ‚Üê comp` is the critical piece. If `comp` fails, the **entire enclosing computation** fails with the same error. Failures propagate automatically ‚Äî similar to exceptions but tracked in the type system.

## Example: Parsing CLI Arguments

Define an error type and functions that can fail:

```lean
inductive Error where
| argumentFormat (arg : String)

def trimDash (arg : String) : Except Error String :=
  if arg.startsWith "--" then return arg.drop 2 |>.toString
  else if arg.startsWith "-" then return arg.drop 1 |>.toString
  else throw (Error.argumentFormat s!"{arg}, expected '-' or '--'")
```

Compose fallible operations with `do`:

```lean
def parseArg (arg : String) : Except Error Argument := do
  let trimmed ‚Üê trimDash arg        -- fails here ‚Üí whole function fails
  match trimmed.splitOn "=" with
  | [name] => return Argument.flag name
  | [name, value] => return Argument.var name value
  | _ => throw (Error.argumentFormat s!"malformed: {trimmed}")
```

## Inspecting Results

Since `Except` is just a data type, you can pattern match on both cases:

```lean
match parseArgs args with
| Except.ok config => -- use config
| Except.error err => -- handle error
```

This is how you cross the boundary between `Except` and `IO` ‚Äî match on the result and decide what to do.

> [!tip] Option vs Except
> `Option Œ±` can express failure but carries **no error information** (just `none`). `Except E Œ±` carries a value of type `E` explaining *what went wrong*. Use `Option` for simple "found/not found"; use `Except` when errors need context.
