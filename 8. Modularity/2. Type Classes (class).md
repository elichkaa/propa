## What They Are

A **type class** is a named collection of operations (an interface). Any type can become a member by providing an **instance** — concrete implementations of those operations.

This is Lean's mechanism for **ad-hoc polymorphism**: the same function name behaves differently depending on the type it's called on.

## Syntax Summary

| Concept | Keyword | Purpose |
|---|---|---|
| Define interface | `class` | Declares required operations |
| Implement interface | `instance` | Provides concrete behavior for a type |
| Require interface | `[ClassName α]` | Constrains a function to types with an instance |

## Key Properties

- **Instances can compute**, not just project fields. An instance for `name` could concatenate fields, return a constant, or do arbitrary logic.
- **Constraints propagate**: if `f` calls `g` and `g` requires `[Serializable α]`, then `f` must also require it (or fix `α` to a concrete type).
- **`variable` scoping**: within a `namespace`, `variable {α} [Serializable α]` applies the constraint to all subsequent definitions automatically.

## Standard Library Examples

Lean's own `Add`, `Mul`, `BEq`, `Repr`, `Inhabited`, `Hashable` are all type classes. Writing `instance : Add Rational` is how you enable `+` for your custom type.

> [!tip]
> Type classes are resolved **at compile time** — there is no runtime dispatch overhead. The compiler fills in the concrete instance wherever it's needed.

## Building Interface Example

````lean
class Building (b : Type) where
  campus   : b → String
  name     : b → String
  address  : b → String
  numFloors: b → Nat
````

- class is the keyword for defining a type class.
- b : Type is the type parameter
	- any type can become a Building by providing an instance.
- You **cannot** call Building.campus "Mensa" directly — you need a concrete instance first.
