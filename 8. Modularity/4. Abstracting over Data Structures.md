## The Idea

Instead of writing an algorithm tied to a specific data structure, **parameterize it over an interface**. This lets you swap implementations without changing the algorithm.

## Pattern

1. **Define a type class** for the data structure (e.g., `Queue`)
2. **Write the algorithm** against the type class (e.g., `bfsWithQueue`)
3. **Provide multiple instances** with different performance characteristics
4. **Instantiate** by picking an implementation at the call site

## Example: Queue Implementations

| Implementation | `enqueue` | `dequeue` | Trade-off |
|---|---|---|---|
| **List** | O(n) | O(1) | Simple, slow enqueue |
| **Banker's Queue** (two lists) | O(1) amortized | O(1) amortized | Fast, slightly more complex |

### Banker's Queue Intuition

Maintain a `front` list (for dequeue) and a `back` list (for enqueue). Enqueue prepends to `back` in O(1). Dequeue takes from `front`. When `front` is empty, reverse `back` — this O(n) cost is amortized across the n enqueues that built it up.

> [!takeaway]
> The algorithm (`bfs`) doesn't know or care which queue it's using. Performance tuning becomes a matter of picking a different type argument — no logic changes.

## Why This Matters in FP

Functional data structures are **immutable** — operations return new values rather than mutating. This means naive implementations (e.g., list append) can be costly. Abstractions let you:
- Start with a simple implementation
- Swap in an optimized one later
- Keep the algorithm code untouched
