# The Expression Problem

## The Dilemma

When a datatype is defined as a single inductive type with multiple variants, **adding a new variant forces changes everywhere** — the type definition itself and every function that matches on it.

This is known as the **Expression Problem**: how do you extend a system with both new types *and* new operations without modifying existing code?

## Two Axes of Extension

|                     | New variants (types)                  | New operations (functions)                      |
| ------------------- | ------------------------------------- | ----------------------------------------------- |
| **Inductive types** | ❌ Hard — must edit type + all matches | ✅ Easy — just add a new function                |
| **Type classes**    | ✅ Easy — just add a new instance      | ❌ Harder — must edit the class or use a new one |

> [!takeaway]
> Neither approach wins on both axes. Type classes solve the "new variant" problem that inductive types struggle with. Choose based on which axis you expect to extend more often.

## In the Lecture

- `BuildingA` as an inductive type required touching every function when a new building type was added.
- Refactoring to a `Building` type class allowed `LaboratoryBuilding` to be added with zero changes to existing code.
